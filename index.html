给技术小白的 Codex 项目协作指南

不会编程，也可以用 Codex 做出复杂系统

你不需要一开始就懂所有代码。你需要学会把目标说清楚，和 Codex 一起拆小任务、做原型、测结果，再把项目稳稳收尾。

技术小白 想做网页 想做工具 想做复杂系统 想补技术基础 想部署和归档

这份指南会教你如何从一个模糊想法开始，和 Codex 一起拆需求、做原型、写代码、测功能、部署上线，并把项目资料整理归档。

01 / Quick Start

快速入门

先记住一个分工：你不用马上变成程序员，但你要负责说清目标、确认方向、验收结果。Codex 负责把事情拆开、写出来、测一遍、讲明白。

你负责

  • 说明这个项目给谁用。
  • 确认第一版先做什么。
  • 判断页面和功能是不是符合真实想法。
  • 打开页面，按清单验收结果。

Codex 负责

  • 把模糊想法拆成任务。
  • 写需求、方案和代码。
  • 跑检查，修问题。
  • 解释改了哪里、为什么这样改。
  • 留下文档和下一步建议。

做网页

重点是内容、布局、视觉和交互。

适合这样问：请先帮我设计页面结构和第一版内容，不要直接写复杂应用。

做工具

重点是输入、处理逻辑、输出和异常情况。

适合这样问：请列出这个工具的输入、处理步骤、输出结果和错误场景。

做复杂系统

重点是用户、权限、数据、流程、测试、部署和维护。

适合这样问：请先拆 MVP、设计数据结构和分阶段实施计划。
新手最容易犯的 5 个错误
错误说法

帮我做个系统。

正确说法

我想做一个客户管理系统，第一版只做客户录入、客户列表和跟进记录。请先帮我拆 MVP。

错误说法

打不开，你修一下。

正确说法

我点击登录后页面空白，地址是本地网页，控制台报错如下。请先定位原因，再做最小范围修复。

错误说法

登录、后台、支付、部署都一起做了吧。

正确说法

请先做第一版核心流程。登录、支付和部署属于高风险功能，后面单独检查。

错误说法

页面看起来可以就行。

正确说法

请同时检查刷新后数据是否还在、手机端能不能看、复制按钮能不能用、有没有测试记录。

错误说法

做完就结束。

正确说法

请同步更新 README、测试记录和阶段总结，告诉我还剩哪些待办。

02 / Workflow

标准工作流

不要跳步骤。复杂系统不是一句话变出来的，而是先定目标、拆第一版、做一点、验一点，慢慢堆稳。

  1. 说清目标先讲给谁用、解决什么问题。
  2. 拆 MVP只保留第一版必须能跑起来的功能。
  3. 写需求文档把目标、范围、验收标准写清楚。
  4. 设计页面和数据先想清页面怎么走、数据放哪里。
  5. 小步开发一次只改一个明确功能。
  6. 测试和修 bug先定位原因，再做最小范围修复。
  7. 部署和归档上线前检查环境，完成后整理资料。
展开：每一步你该怎么问 Codex

说清目标

请先帮我澄清目标用户、核心问题、第一版范围和验收标准。

拆 MVP

请把这个系统拆成 V1、V2、V3，并说明第一版暂时不做什么。

写需求文档

请把已经确认的内容写成需求文档，包含目标、范围、功能和验收标准。

设计页面和数据

请设计页面结构、用户流程、数据字段，以及它们之间的对应关系。

小步开发

请一次只实现一个小任务，完成后告诉我改了什么、怎么运行、怎么验证。

测试和修 bug

请先定位原因，再做最小范围修复，并记录验证结果。

03 / Basics

技术基础

你不需要先学完整套编程课。先补一点项目常识：网页由哪些文件组成，样式在哪里改，按钮为什么会动，最后要怎么在浏览器里验收。

第一批：网页基础。先看懂 HTML、CSS、JavaScript、本地服务和浏览器验收。等你开始做需要保存数据的工具或系统，再补后端、数据库、接口、权限和安全。
HTML：网页骨架

HTML 决定网页里有什么内容，比如标题、段落、导航、卡片和按钮。

CSS：网页外观

CSS 决定颜色、字号、间距、卡片、两栏布局和手机端适配。

对应当前项目

知道样式错乱、文字挤压、手机端溢出，通常要检查 CSS 布局。
可以这样问
请根据截图定位这个排版问题，先说明原因，再做最小 CSS 修改。
小练习

JavaScript：网页交互

JavaScript 负责复制按钮、导航高亮、弹窗、切换等用户操作后的反应。

对应当前项目
02_网页应用/prototype/script.js
学到什么程度
知道按钮写在 HTML 里不等于一定能工作，真正行为通常在 JavaScript 里。
可以这样问
请解释复制按钮从点击到显示“已复制”的完整过程。如果复制失败，可能有哪些原因？
小练习
试着点击复制按钮，看按钮文字有没有变化。
本地服务：更接近真实运行环境

本地服务是在自己电脑上临时启动一个网页地址，让浏览器像访问网站一样访问项目。

对应当前项目
02_网页应用/prototype/local-server.mjs
学到什么程度
知道直接双击 HTML 和通过本地服务访问不完全一样，本地服务不等于正式上线。
可以这样问
请解释 `node local-server.mjs 8010` 每一部分是什么意思。如果网页打不开，我该检查哪里？
小练习
试着用本地服务打开页面：http://127.0.0.1:8010/index.html。
浏览器：最终验收的地方

代码写对不等于页面看起来一定对，最终还要在浏览器里检查真实效果。

当前项目要检查
页面能否打开、中文是否乱码、导航是否跳转、复制按钮是否工作、手机宽度是否溢出。
学到什么程度
知道截图、地址、复现步骤和控制台报错，都能帮助 Codex 快速定位问题。
可以这样问
我看到页面这里排版不对，截图如下。请判断可能是哪类问题，并告诉我需要检查哪些文件。
小练习
试着把页面缩窄到手机宽度，看看有没有文字溢出。

下一批：系统基础

前端、后端、数据库、API、HTTP 状态码和数据流。等你开始做需要保存数据的工具或系统时再展开。

再下一批：工程、上线和安全

命令行、Git、测试、构建、部署、环境变量、密钥、权限、备份和回滚。

展开：系统基础先知道这些

系统基础不是让你马上学会后端开发，而是让你知道一个系统为什么需要保存数据、为什么会分前端和后端、页面和服务器之间怎么说话。

前端
你在浏览器里看到和操作的部分，比如页面、按钮、表单和列表。
后端
在服务器上处理规则、保存数据、检查权限的部分。用户通常看不见它，但系统离不开它。
数据库
专门保存长期数据的地方，比如用户、订单、项目、记录和设置。
API
前端向后端要数据、提交数据时使用的约定入口，可以先理解成“页面和服务器之间的接口”。
HTTP 状态码
浏览器和服务器沟通后的结果提示，比如 200 表示成功，404 表示找不到，500 表示服务器出错。
数据流
一条数据从用户输入，到前端提交，再到后端处理、数据库保存，最后回到页面展示的完整路径。
什么时候学
当你的项目开始需要登录、保存记录、多人使用或长期维护时，就该让 Codex 帮你补这一批基础。
可以这样问
我是技术小白，请结合当前项目解释前端、后端、数据库、API 和数据流分别是什么，并说明第一版是否真的需要它们。
展开：工程、上线和安全先知道这些

工程基础关注的是“项目怎么稳定地做完、上线、保护和回退”。它不是炫技，而是避免项目上线后打不开、丢数据或泄露密钥。

命令行
用文字命令操作项目的地方，常用于启动服务、安装依赖、运行检查和发布项目。
Git
记录代码变化的工具。它能让你知道改了什么，也方便在出错时回到旧版本。
测试
用清单或自动化方式确认功能真的能用，不只看页面“好像没问题”。
构建
把开发中的代码整理成适合发布的文件。静态网页可能很简单，复杂应用通常必须构建。
部署
把项目放到别人能访问的线上地址，例如 GitHub Pages、服务器或云平台。
环境变量
不适合写死在代码里的配置，比如服务器地址、数据库连接信息和密钥。
密钥和权限
密钥相当于系统通行证，不能放进公开代码；权限决定谁能看、改、删除哪些数据。
备份和回滚
备份是出问题前留下退路，回滚是上线失败时恢复到上一个可用版本。
可以这样问
请帮我做上线前检查：命令、Git 状态、构建、环境变量、密钥、权限、备份和回滚分别要确认什么？

05 / Systems & Case

复杂系统与案例

复杂系统不要一口气做完。先做主数据，再加详情、流程、同步、记录、权限和部署。每一步都能验收，项目才不容易失控。

复杂系统应该怎么拆
01

项目池

先把项目名称、目标、状态和优先级记下来。

02

项目详情

再看每个项目的背景、需求、进度和文档。

03

进展图

把状态画出来，一眼看到卡在哪里。

04

Codex 同步

把外部会话和项目关联，但不让它取代主数据。

05

项目轨迹

记录每次推进，方便回头看决策。

06

每日简报

把当天进展沉淀成短摘要。

07

登录保护

等核心流程稳定后，再处理账号和权限。

08

部署备份

上线前检查环境、日志、备份和回滚方案。

完整案例：个人项目管理系统
  1. 最初想法记录想法、需求和项目进度。
  2. 关键转折项目池是主数据，Codex 项目是外部来源。
  3. 原型阶段先做项目进度驾驶舱。
  4. 功能阶段逐步加入项目池、进展图、详情、同步、轨迹和简报。
  5. 部署阶段准备服务器部署方案。
  6. 归档阶段整理项目文档、源码、数据和主会话。

06 / Ship

验收、部署与归档

代码写完不等于项目结束。你还要确认它能打开、能使用、能解释、能回看，并且没有把密码、密钥或无关源码放进归档。

验收检查清单
  • Codex 有没有说明改了哪些文件？
  • 有没有说明实现了什么？
  • 有没有告诉我怎么运行？
  • 有没有告诉我怎么验证？
  • 有没有运行测试，或者说明为什么没跑？
  • 有没有运行构建或基础检查？
  • 有没有解释 bug 原因？
  • 有没有说明哪些事情还没做？
  • 有没有提醒风险和后续待办？
部署和归档

部署前检查

  • 构建是否通过。
  • 本地是否能打开。
  • 手机端是否能正常阅读。
  • 密码和密钥是否没有写死。
  • 关键功能是否已经验证。

应该归档什么

  • 需求文档、设计方案和实施计划。
  • 源码、测试记录和部署说明。
  • 操作记录、阶段总结和待办清单。
  • 重要数据的备份说明。
不要归档：账号密码、API Key、其他项目源码、无关缓存和依赖目录。
