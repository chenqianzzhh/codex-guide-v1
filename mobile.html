01 / 快速入门

先记住这个分工

你不用马上变成程序员，但你要负责说清目标、确认方向、验收结果。Codex 负责拆开、写出来、测一遍、讲明白。

你负责

  • 说明项目给谁用。
  • 确认第一版先做什么。
  • 判断结果是否符合真实想法。
  • 按清单验收页面和功能。

Codex 负责

  • 把模糊想法拆成任务。
  • 写需求、方案和代码。
  • 跑检查，修问题。
  • 留下文档和下一步建议。
新手最容易犯的 5 个错误

不要说：帮我做个系统。
可以说：我想做客户管理系统，第一版只做客户录入、客户列表和跟进记录，请先帮我拆 MVP。

不要说：打不开，你修一下。
可以说：我点击登录后页面空白，控制台报错如下。请先定位原因，再做最小范围修复。

不要说：登录、后台、支付、部署都一起做。
可以说：请先做第一版核心流程，高风险功能后面单独检查。

不要说：页面看起来可以就行。
可以说：请同时检查手机端、复制按钮、刷新后数据和测试记录。

不要说：做完就结束。
可以说：请同步更新 README、测试记录和阶段总结。

02 / 标准工作流

不要跳步骤

复杂系统不是一句话变出来的。先定目标、拆第一版、做一点、验一点，慢慢堆稳。

  1. 说清目标给谁用，解决什么问题。
  2. 拆 MVP只保留第一版必须能跑起来的功能。
  3. 写需求文档把目标、范围、验收标准写清楚。
  4. 设计页面和数据先想清页面怎么走、数据放哪里。
  5. 小步开发一次只改一个明确功能。
  6. 测试和修 bug先定位原因，再最小范围修复。
  7. 部署和归档上线前检查环境，完成后整理资料。

03 / 技术基础

先补项目常识

不用先学完整编程课。先知道网页由哪些文件组成，样式在哪里改，按钮为什么会动，最后怎么在浏览器里验收。

HTML

网页骨架，对应 index.html。

CSS

页面外观，对应 styles.css。

JavaScript

按钮和交互，对应 script.js。

本地服务

更接近真实访问，对应 local-server.mjs。

浏览器

最终验收页面、按钮、手机宽度和报错。

下一批：系统基础

前端：你在浏览器里看到和操作的页面。

后端：在服务器上处理规则、保存数据和检查权限的部分。

数据库：长期保存用户、项目、订单、记录等数据的地方。

API：页面向服务器要数据、提交数据时使用的入口。

HTTP 状态码：浏览器和服务器沟通后的结果提示，比如 200、404、500。

数据流：数据从输入、提交、保存到展示的完整路径。

再下一批：工程、上线和安全

命令行：用文字命令启动服务、运行检查和管理项目。

Git：记录代码变化，方便知道改了什么和回到旧版本。

测试和构建：上线前确认功能能用，并整理出适合发布的文件。

部署：把项目放到别人能访问的线上地址。

环境变量和密钥：保存不适合写进公开代码的配置和通行证。

权限、备份和回滚：控制谁能做什么，并给项目留下出问题时的退路。

05 / 复杂系统与案例

复杂系统要分阶段做

先做主数据，再加详情、流程、同步、记录、权限和部署。每一步都能验收，项目才不容易失控。

01

项目池：先把项目名称、目标、状态和优先级记下来。

02

项目详情：再看每个项目的背景、需求、进度和文档。

03

进展图：把状态画出来，一眼看到卡在哪里。

04

Codex 同步：把外部会话和项目关联，但不让它取代主数据。

05

项目轨迹：记录每次推进，方便回头看决策。

06

每日简报：把当天进展沉淀成短摘要。

07

登录保护：等核心流程稳定后，再处理账号和权限。

08

部署备份：上线前检查环境、日志、备份和回滚方案。

06 / 验收部署归档

写完代码后要收口

确认它能打开、能使用、能解释、能回看，并且没有把密码、密钥或无关源码放进归档。

  • Codex 有没有说明改了哪些文件？
  • 有没有说明实现了什么？
  • 有没有告诉我怎么运行？
  • 有没有告诉我怎么验证？
  • 有没有运行测试，或者说明为什么没跑？
  • 有没有提醒风险和后续待办？

部署前

  • 本地能打开。
  • 手机端能阅读。
  • 密钥没有写死。
  • 关键功能已验证。

归档时

  • 保存需求、方案和源码。
  • 保存测试记录和部署说明。
  • 保存阶段总结和待办。
  • 不要保存密码、API Key 和无关源码。
